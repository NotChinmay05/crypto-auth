Build a JWT-inspired authentication service built entirely in Python where you implement the cryptographic primitives from scratch without any external crypto libraries. 
The token — called a CAT (CryptoAuth Token) — follows the three-part dot-separated structure of JWT: a base64url-encoded header containing the algorithm name and token type, a base64url-encoded payload containing user-supplied claims plus server-generated fields like issued-at time, expiration time, and a unique token ID called jti, and a base64url-encoded signature that binds the two together. The payload is encrypted using AES-128-CBC which you implement from scratch including the full S-box substitution, key schedule expansion, AddRoundKey, ShiftRows, and MixColumns steps along with PKCS7 padding and a randomly generated IV prepended to the ciphertext, so the payload is never stored or transmitted in plaintext. The signature is computed using HMAC-SHA256 where you also implement the SHA-256 compression function from scratch including the message schedule, the eight working variables, and the 64 rounds of bitwise operations, then wrap it in the HMAC construction using inner and outer padding with the server's secret key, signing the concatenation of the encoded header and encoded payload so any tampering to either part invalidates the signature. When a user registers, their password is hashed using SHA-256 with a random salt and stored, and the raw password is never persisted. When they log in, the server verifies the password hash, builds the payload with their user ID and any custom claims, encrypts it with AES-CBC, signs the result with HMAC-SHA256, and returns the final dot-joined token. On verification, the server first checks the HMAC signature to confirm integrity and authenticity, then checks the jti against an in-memory blacklist to handle revocation, then checks the expiry timestamp, and only then decrypts the payload to return the claims — this ordering is important because you never decrypt an unauthenticated ciphertext. 
The FastAPI layer sits on top of this pure-Python crypto core and exposes endpoints for register, login, verify, refresh, revoke, and inspect, where inspect decodes the token structure without verifying it to show the raw internals for educational purposes. The backend has eight endpoints built on FastAPI. The first is POST /auth/register which accepts a username, password, and optional custom claims like role or user metadata, generates a random 16-byte salt, computes SHA-256 of the salt concatenated with the password entirely in pure Python, and stores the username, salt, and password hash in the database alongside the custom claims, returning a success response without ever persisting the raw password. The second is POST /auth/login which accepts a username and password, retrieves the stored salt and hash for that user, recomputes the SHA-256 hash of the provided password with the stored salt, compares it in constant time to prevent timing attacks, and if it matches builds the CAT token by constructing the header and payload dictionaries, serializing and base64url-encoding them, encrypting the payload with AES-128-CBC using a randomly generated IV and the server's master encryption key, computing HMAC-SHA256 over the concatenation of the encoded header and encrypted encoded payload using the server's signing key, joining all three parts with dots, and returning the token to the client. The third is POST /auth/verify which accepts a token, splits it into its three parts, recomputes the HMAC-SHA256 of the header and payload parts using the server's signing key and compares it in constant time against the provided signature rejecting the token immediately if it does not match, then checks the jti field against the in-memory revocation blacklist rejecting if found, then decodes and decrypts the payload using AES-128-CBC to extract the claims, checks the expiration timestamp, and if all checks pass returns the decrypted claims to the caller. The fourth is POST /auth/refresh which accepts a valid non-expired token, runs it through the full verification pipeline, and if valid issues a brand new token with a fresh jti, a new issued-at timestamp, a new expiration time, and a new IV for AES encryption while adding the old token's jti to the revocation blacklist so the old token cannot be reused, effectively implementing single-use refresh semantics. The fifth is POST /auth/revoke which accepts a token and an optional reason, verifies the signature to confirm the token is genuine and was issued by this server, extracts the jti from the payload, and adds it to the in-memory blacklist so that token is permanently rejected on all future verify calls even though its cryptographic signature remains mathematically valid, demonstrating the separation between cryptographic validity and authorization state. The sixth is GET /auth/inspect which accepts a token as a query parameter and decodes and displays its structure without performing any signature verification or decryption, showing the raw base64url-decoded header as a readable JSON object, the raw encrypted payload bytes as a hex dump, and the raw signature bytes, intended purely for educational and debugging purposes to let users see inside the token format. The seventh is GET /auth/me which accepts a token in the Authorization header, runs full verification, and returns the authenticated user's stored profile including their username and any custom claims they registered with, serving as the canonical protected resource endpoint that demonstrates the full auth flow where a client must present a valid token to access user data. The eighth is GET /health which returns the server status, the current timestamp, and the count of currently blacklisted tokens, serving as a simple liveness check and also useful in the frontend dashboard to show the server is running and how many tokens have been revoked in the current session.
The frontend has four panels — a Token Playground where you register and log in and see the token broken into its three color-coded parts with decoded header and hex-dumped encrypted payload, a Protected Resource panel that makes a live API call succeeding or failing based on token validity, a Token Inspector where you paste any token and see its decoded structure and expiry countdown, and a Revocation Demo that shows a token failing verification after revocation even though its cryptographic signature is still mathematically valid — demonstrating that cryptographic validity and authorization are separate concerns.

Build a steganographic image signing system built entirely in Python where you embed a cryptographic signature invisibly inside the image pixels themselves so the image remains visually identical but carries a tamper-evident seal that can be extracted and verified later. When a user uploads an image for signing, the server first reads the raw pixel data as a flat array of RGB byte values using only the standard Python PIL library for pixel access, then computes a SHA-256 hash of the original pixel data implemented from scratch in pure Python, then constructs a certificate dictionary containing the SHA-256 hash, the author name, a timestamp, and a unique image ID, then serializes this certificate to a compact JSON string, then computes an HMAC-SHA256 of the serialized certificate using the server's secret signing key to produce the final signature, then concatenates a fixed magic header bytes sequence with the length of the certificate as a 4-byte big-endian integer followed by the certificate JSON bytes and the HMAC signature bytes to form the complete payload that will be hidden inside the image. The embedding works using LSB steganography where you take each byte of the payload and spread its 8 bits across 8 consecutive pixels by replacing the least significant bit of the red channel of each pixel with one bit of the payload, which changes each affected pixel's red value by at most 1 out of 255 making the difference completely invisible to the human eye, and after embedding you reconstruct the full image with these minimally modified pixels and return it to the user as a downloadable PNG so no compression artifacts destroy the hidden bits. When another user uploads an image for verification, the server reads the pixel data, extracts the LSBs of the red channel of the first N pixels where N is determined by first reading the length prefix from the first 40 pixels, reconstructs the payload bytes from these bits, parses the magic header to confirm this is a signed image, extracts the certificate JSON and the HMAC signature, recomputes HMAC-SHA256 of the certificate using the server's signing key and compares it in constant time to verify the certificate itself was not forged, then recomputes SHA-256 of all pixel data excluding the LSB-carrier pixels which are the ones that were modified during signing, compares this recomputed hash against the hash stored inside the certificate, and if both the HMAC check and the pixel hash check pass returns an AUTHENTIC result with the certificate details showing the author, timestamp, and image ID, but if either check fails returns a TAMPERED result and additionally runs a pixel-by-pixel diff between the hash-expected state and the current state to highlight which regions of the image were modified, displayed as a red overlay mask showing the tampered areas. The FastAPI layer exposes three endpoints for this module — POST /image/sign which accepts a multipart image upload and optional author metadata and returns the signed image file, POST /image/verify which accepts a multipart image upload and returns the verification result with certificate details or tamper evidence, and GET /image/inspect which extracts and displays the embedded certificate and signature bytes in raw form without performing verification for educational purposes. The frontend for this module has a two-panel layout where the left panel is the signing station where you drag and drop an image, optionally enter your name, click sign, and download the returned image which looks pixel-perfect identical to the original, and the right panel is the verification station where you upload any image signed or potentially tampered and see either a green authentic badge showing the embedded certificate details or a red tampered alert showing the diff overlay highlighting exactly which pixels were changed, making it visually compelling and immediately understandable as a demonstration of how cryptographic hashing detects even a single pixel modification anywhere in the image.
